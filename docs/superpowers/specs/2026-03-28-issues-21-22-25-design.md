## #21 — Cross-PDS Attestation Verification

### Problem

`verifyAttestation` only checks local `records_index`. For federation, a PDS must verify attestations issued by communities on other PDS instances.

### Approach

Enhance `verifyAttestation` to support remote community DIDs. When the community isn't local, resolve the DID to its home PDS and fetch the attestation via the remote PDS's `com.atproto.repo.getRecord` endpoint.

No CAR fetching or MST signature verification for MVP. We trust the remote PDS's `getRecord` response — same trust model as how ATProto clients trust PDS responses. Full cryptographic verification can be added later.

### New Module

`src/federation/remote-verify.ts`:
- `resolveDidToPds(did)` — resolves a DID to its PDS service URL by fetching the DID document (PLC directory for `did:plc`, `/.well-known/did.json` for `did:web`)
- `fetchRemoteRecord(pdsUrl, did, collection, rkey)` — fetches a record from a remote PDS via `com.atproto.repo.getRecord`
- In-memory TTL cache (5 min) for remote results

### Changes to verifyAttestation

Current flow: query local `records_index` → return result.

New flow:
1. Query local `records_index`
2. If found locally → return as before
3. If not found → check if the community DID exists locally (`communities` table)
4. If community is local but attestation not found → return `valid: false` (not remote, just doesn't exist)
5. If community is NOT local → call `resolveDidToPds()` + `fetchRemoteRecord()`
6. Return result with `remote: true` flag and `pdsUrl` for transparency

### Response (remote case)

```json
{
  "valid": true,
  "remote": true,
  "pdsUrl": "https://other-pds.example.com",
  "attestation": { ... },
  "communityDid": "did:plc:remote123",
  "communityHandle": null,
  "expired": false
}
```

| File | Responsibility |
|------|----------------|
| **Create:** `src/federation/remote-verify.ts` | DID resolution + remote record fetching + cache |
| **Modify:** `src/api/net.openfederation.community.verifyAttestation.ts` | Add remote verification fallback |

## #22 — Matrix Bridge

### Architecture

Standalone service in `bridges/matrix/` with its own `package.json`. Runs as a separate process. One-way sync: PDS → Matrix. Syncs community membership and roles to Matrix Spaces.

### Separation of Concerns

- **PDS owns:** users, roles, permissions, attestations
- **Matrix owns:** channel structure, per-channel permissions
- **Bridge syncs:** membership + roles from PDS → Matrix Space power levels

The bridge does not manage channels. Space admins configure per-channel access in Matrix using Matrix's own tools.

### Three Deployment Modes

**1. Public Matrix (`mode: "public"`)**
- Users link existing Matrix IDs via custom profile collection (`app.matrix.actor.profile` with `{ matrixId: "@alice:matrix.org" }`)
- Bridge reads Matrix IDs from user profiles via `getProfile` endpoint
- Bridge invites/kicks users from the Matrix Space
- Friction: users must register on both platforms separately

**2. Community-owned server (`mode: "self-hosted"`)**
- Community runs their own Matrix homeserver
- Bridge auto-provisions Matrix accounts from PDS handles via Matrix admin API
- Handle template: `{handle}:club.hackney.org`
- Bridge creates accounts on join, deactivates on leave

**3. Partner-hosted (`mode: "partner-hosted"`)**
- Partner (e.g., Grvty) operates Matrix infrastructure for multiple communities
- Shared homeserver (`matrix.grvty.net`) or dedicated per-community
- Bridge uses partner admin API to provision accounts and spaces
- Communities get Matrix without ops burden
- Maps to existing Partner API relationship

### Sync Flow

1. Bridge authenticates to PDS (session token or partner key)
2. Polls community member list on interval (configurable, default 60s)
3. For each configured `communityDid → matrixSpaceId` mapping:
   - Fetch PDS members via `listMembers`
   - Fetch member roles via role records
   - Compare with Matrix Space membership
   - Sync: invite new members, kick removed members, update power levels
4. Log sync events

### Bridge Config

```json
{
  "pdsUrl": "https://pds.openfederation.net",
  "pdsAuth": {
    "handle": "bridge-bot",
    "password": "..."
  },
  "mode": "partner-hosted",
  "matrix": {
    "homeserverUrl": "https://matrix.grvty.net",
    "adminToken": "syt_...",
    "handleTemplate": "{handle}:grvty.net"
  },
  "communityMappings": [
    {
      "communityDid": "did:plc:abc123",
      "matrixSpaceId": "!space:grvty.net",
      "rolePowerLevels": {
        "owner": 100,
        "moderator": 50,
        "coach": 25,
        "member": 0
      }
    }
  ],
  "syncIntervalMs": 60000
}
```

### Dependencies

- `matrix-js-sdk` — official Matrix client SDK
- Node.js `fetch` for PDS XRPC calls (no SDK dependency — the bridge is a reference implementation)

### Files

| File | Responsibility |
|------|----------------|
| **Create:** `bridges/matrix/package.json` | Package manifest |
| **Create:** `bridges/matrix/tsconfig.json` | TypeScript config |

### What's NOT Included

- Channel management (Matrix-side concern)
- Message bridging (out of scope)
- Matrix → PDS sync (one-way only)
- Web UI for bridge configuration

## #25 — CLI Expansion

### Approach

Add commands to existing `cli/pds-cli.ts`. Each command wraps an XRPC call using the existing CLI infrastructure (session management, HTTP client, output formatting).

### New Commands

**Community Management:**
- `list-communities` — list public communities
- `update-community --did <did> [--displayName] [--description] [--visibility] [--joinPolicy]`
- `delete-community --did <did>`
- `suspend-community --did <did>`
- `export-community --did <did>`

**Member Management:**
- `list-members --did <did>`
- `kick-member --did <did> --memberDid <memberDid>`
- `update-role --did <did> --memberDid <memberDid> --roleRkey <rkey>`
- `list-roles --did <did>`
- `create-role --did <did> --name <name> --permissions <p1,p2,...>`

**Attestations:**
- `issue-attestation --did <did> --subjectDid <subjectDid> --type <type> --claim <json>`
- `verify-attestation --did <did> --rkey <rkey>`
- `list-attestations --did <did> [--subjectDid]`
- `delete-attestation --did <did> --rkey <rkey>`

**External Keys:**
- `set-key --rkey <rkey> --type <type> --purpose <purpose> --publicKey <key>`
- `list-keys --did <did>`
- `delete-key --rkey <rkey>`
- `resolve-key --publicKey <key>`

**Partner Keys:**
- `create-partner-key --name <name> [--origin <origin>]`
- `list-partner-keys`
- `revoke-partner-key --id <id>`

**Governance:**
- `set-governance --did <did> --model <model> [--quorum <n>] [--voterRole <role>]`
- `create-proposal --did <did> --collection <col> --rkey <rkey> --action <write|delete> [--record <json>]`
- `vote --did <did> --proposalRkey <rkey> --vote <for|against>`
- `list-proposals --did <did> [--status <status>]`

**Profiles:**
- `get-profile --did <did>`
- `update-profile [--displayName] [--description]`

### Pattern

Each command follows the existing CLI pattern:
```typescript
.command('list-members')
  .description('List members of a community')
  .requiredOption('--did <did>', 'Community DID')
  .action(async (opts) => {
    const session = loadSession();
    const res = await xrpcGet(session, 'net.openfederation.community.listMembers', { communityDid: opts.did });
    console.table(res.members.map(m => ({ did: m.did, handle: m.handle, role: m.role })));
  });
```

### Files

| File | Responsibility |
|------|----------------|
| **Modify:** `cli/pds-cli.ts` | Add all new commands |

The file will grow significantly (~30 commands). If it becomes unwieldy, commands can be split into separate files by domain (community, attestation, governance) and imported. But for MVP, one file is fine — it's all mechanical XRPC wrappers.
